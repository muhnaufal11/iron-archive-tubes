## A. Black Box Manual — Equivalence Partitioning (EP)

### A.1 Aturan validasi yang diuji
Form Personel — `UserController@store` (`app/Http/Controllers/UserController.php:44`):

| Field | Aturan |
|-------|--------|
| `name` | required, string, max:255 |
| `email` | required, email, max:255, unique:users |
| `password` | required, min:8, confirmed |
| `role` | required, in:admin,user |

Registrasi — `RegisterController@validator`: `name` (required,max:255), `email` (required,email,max:255,unique), `password` (required,min:8,confirmed).

### A.2 Identifikasi Kelas Ekuivalensi

| Field | Kelas Valid | Kelas Tidak Valid |
|-------|-------------|-------------------|
| name | V1: 1–255 karakter | I1: kosong |
| email | V2: format benar & belum terdaftar | I2: format salah · I3: sudah terdaftar (duplikat) |
| password | V3: ≥ 8 karakter & konfirmasi cocok | I4: < 8 karakter · I5: konfirmasi tidak cocok |
| role | V4: "admin" / "user" | I6: di luar {admin, user} |

### A.3 Tabel Test Case EP

| ID | Input yang diuji | Kelas | Hasil Diharapkan | Diimplementasikan di test() |
|----|------------------|-------|------------------|------------------------------|
| EP-01 | Semua field valid (role=user) | V1–V4 | Diterima, user tersimpan | `EP valid: admin merekrut personel baru` |
| EP-02 | email = `bukan-email` | I2 | Ditolak (error email) | `EP invalid: format email salah ditolak` |
| EP-03 | email sudah ada di DB | I3 | Ditolak (error unique) | `EP invalid: email duplikat ditolak (unique)` |
| EP-04 | role = `superadmin` | I6 | Ditolak (error role) | `EP invalid: role di luar enum ditolak` |
| EP-05 | password ≠ konfirmasi | I5 | Ditolak (error password) | `EP invalid: konfirmasi password tidak cocok ditolak` |
| EP-06 | Registrasi data valid | V1–V3 | Sukses, user tersimpan & login | `EP valid: registrasi publik berhasil & user tersimpan` |

## B. Black Box Manual — Boundary Value Analysis (BVA)

Fokus pada batas numerik/panjang: **password `min:8`** dan **`max:255`**.

### B.1 Boundary password (`min:8`)

| ID | Nilai uji | Posisi batas | Hasil Diharapkan | Implementasi |
|----|-----------|--------------|------------------|--------------|
| BVA-01 | 7 karakter (`rahasi7`) | batas bawah − 1 | **Ditolak** | `BVA: password 7 karakter ditolak` |
| BVA-02 | 8 karakter (`rahasia8`) | batas bawah | **Diterima** | `BVA: password 8 karakter diterima` |
| BVA-03 | 9 karakter (`rahasia12`) | batas bawah + 1 | Diterima | *(tambahan — bisa ditambahkan ke test)* |
| BVA-04 | Registrasi password 7 char | batas bawah − 1 | **Ditolak** | `BVA: registrasi dengan password 7 karakter ditolak` |

### B.2 Boundary panjang `max:255` (analisis tambahan untuk name/email)

| ID | Nilai uji | Posisi batas | Hasil Diharapkan |
|----|-----------|--------------|------------------|
| BVA-05 | name 255 karakter | batas atas | Diterima |
| BVA-06 | name 256 karakter | batas atas + 1 | Ditolak |
| BVA-07 | email 255 karakter | batas atas | Diterima |
| BVA-08 | email 256 karakter | batas atas + 1 | Ditolak |

> BVA-05–08 adalah analisis manual; boundary `max:255` sudah dibuktikan terimplementasi pada field `name` kategori (lihat bagian Nadhif), dan dapat ditambahkan untuk field user sebagai pelengkap.

## C. Black Box Manual — State Transition (Autentikasi)

### C.1 Daftar State
- **S0 — Tamu (Guest):** belum login.
- **S1 — Terautentikasi:** sudah login (akses penuh).

### C.2 Diagram State Transition

```
                login valid / registrasi valid
        ┌───────────────────────────────────────────────┐
        │                                                 ▼
   ┌─────────┐                                      ┌──────────────────┐
   │ S0 Tamu │ ◄────────────── logout ───────────── │ S1 Terautentikasi │
   └─────────┘                                      └──────────────────┘
     ▲     │
     │     │  login invalid  /  akses halaman terproteksi
     └─────┘  (tetap di S0, muncul error / redirect ke login)
```

### C.3 Tabel State Transition

| State Awal | Event (Transisi) | State Akhir | Output / Reaksi Sistem |
|------------|------------------|-------------|------------------------|
| S0 Tamu | Akses halaman terproteksi (`/vehicles`) | S0 Tamu | Redirect ke `/login` |
| S0 Tamu | Login kredensial **valid** | S1 Terautentikasi | Redirect ke `/home` |
| S0 Tamu | Login kredensial **salah** | S0 Tamu | Tetap di login + pesan error |
| S0 Tamu | Registrasi **valid** | S1 Terautentikasi | Akun dibuat + redirect `/home` |
| S1 Terautentikasi | Logout | S0 Tamu | Redirect ke halaman login |

### C.4 Pemetaan ke Test (1 transisi = 1 skenario)

| Transisi | Implementasi di test() |
|----------|------------------------|
| Akses terproteksi → login | `transisi: guest akses halaman terproteksi diarahkan ke login` |
| Login valid → authenticated | `transisi: login valid memindahkan ke state authenticated` |
| Login invalid → tetap guest | `transisi: login invalid tetap di state guest` |
| Logout → guest | `transisi: logout mengembalikan ke state guest` |
| Registrasi valid → authenticated | `transisi: registrasi valid langsung authenticated` |

> Catatan: transisi yang sama juga diuji di level UI oleh Selenium (`tests/Selenium/auth_test.php`) — sesuai brief, State Transition diimplementasikan ke PEST Feature **dan** Selenium.

## D. Bab 4 — Implementasi PEST

### D.1 Tentang PEST
PEST adalah testing framework untuk PHP yang berjalan di atas PHPUnit dengan sintaks `test('deskripsi', function () { ... })` yang lebih ringkas dan mudah dibaca. Dipakai untuk mengimplementasikan **Unit Test** (hasil Basis Path Testing) dan **Feature Test** (hasil EP/BVA & State Transition).

### D.2 Setup Lingkungan
```bash
composer require pestphp/pest pestphp/pest-plugin-laravel --dev --with-all-dependencies
```
- Database pengujian memakai **SQLite `:memory:`** (dikonfigurasi di `phpunit.xml`) sehingga cepat dan tidak menyentuh data asli.
- File `tests/Pest.php` mengaktifkan trait **`RefreshDatabase`** (migrasi dijalankan otomatis tiap test) dan menyediakan helper `admin()` / `prajurit()` untuk membuat user.

### D.3 Struktur Folder & Pemetaan Teknik
| Folder | Teknik manual | Jenis test |
|--------|---------------|------------|
| `tests/Unit/` | Basis Path Testing (White Box) | PEST Unit Test |
| `tests/Feature/` | EP + BVA & State Transition (Black Box) | PEST Feature Test |

### D.4 Cara Menjalankan
```bash
php artisan test                                       # semua test
php artisan test tests/Feature/UserAuthValidationTest.php   # EP/BVA Auth (bagian Naufal)
php artisan test tests/Feature/AuthStateTransitionTest.php  # State Transition (bagian Naufal)
```

### D.6 Analisis Singkat
Seluruh partisi valid diterima dan partisi tidak valid ditolak sesuai harapan, membuktikan aturan validasi (`required`, `email`, `unique`, `min:8`, `confirmed`, `in:admin,user`) berfungsi benar. Pengujian State Transition memastikan perpindahan state autentikasi (Tamu ↔ Terautentikasi) berjalan benar termasuk penanganan login gagal dan proteksi halaman.
